# Brief Hardening Spec

## Summary

Brief should get more opinionated, more boring, and more honest.

The core idea is right:
- markdown convention first
- tiny CLI
- agent does the thinking

The failure mode is also clear:
- a workspace can be legacy, stale, or half-broken and still look vaguely successful
- agents then trust Brief too much
- that turns Brief into fake confidence theater

This spec makes Brief trustworthy by enforcing one canonical schema, explicit health states, clean degradation, and a real migration path.

## Product Position

Brief is not a planner app.
Brief is not a hidden reasoning engine.
Brief is a markdown convention plus a small CLI that helps agents fetch state and verify trust.

The source of truth should be:
- `.brief/` files
- `rules/` markdown
- `raw/` fetched inputs

The CLI should stay small:
- `brief init`
- `brief fetch`
- `brief check`
- `brief migrate` (new, justified by schema churn)

Anything beyond that should be treated as suspect.

## Problems

### 1. Schema ambiguity
A workspace can be old, partial, or malformed, and the CLI may still appear to work.

That makes it hard for agents to answer the only question that matters first:

**Can I trust this Brief workspace as a current steering layer?**

### 2. Success signals are too soft
A command succeeding is not the same thing as:
- current schema present
- rules available
- enrichment path valid
- priorities artifact trustworthy

### 3. Degradation is too magical
Legacy or broken workspaces should degrade clearly.
They should not half-work and force the agent to guess.

### 4. No first-class migration path
If Brief is going to evolve, it needs a supported migration path.
Otherwise every schema change creates a swamp of almost-compatible workspaces.

### 5. Too much trust in CLI messaging
The CLI should not imply capabilities that only exist as convention or no longer exist at all.

## Goals

1. One canonical current schema.
2. Explicit health states before trust.
3. Graceful degradation for legacy, stale, missing, and broken workspaces.
4. Machine-readable health for automation.
5. Convention-first behavior for enrichment and planning.
6. A supported migration path for older workspaces.
7. Keep the CLI small.

## Non-goals

- Do not bring back a fat command surface.
- Do not move planning logic into the CLI.
- Do not hide schema problems behind best-effort success.
- Do not make Brief a hard dependency for every automation flow.

## Core Product Decisions

## A. Current schema is canonical

The startup template generated by `brief init --template startup` is the canonical schema.

Health detection should be measured against that schema, not against ad hoc path checks scattered across commands.

### Canonical current schema includes
Required directories:
- `.brief/rules/`
- `.brief/raw/`

Required artifacts:
- `.brief/priorities.md`
- `.brief/PRIORITIES-HUMAN.md`
- `.brief/rules/BUILD.md`
- `.brief/rules/INTERVIEW.md`
- any other files the startup template declares required

Implementation detail:
- keep an internal manifest of required files/dirs derived from init templates
- future template changes update that manifest in one place

## B. Health is first-class

Brief should expose these health states:
- `healthy-current-schema`
- `legacy-schema`
- `missing`
- `misconfigured`
- `stale`

### Meaning

#### `healthy-current-schema`
The workspace matches the current schema and is safe to treat as a steering layer.

#### `legacy-schema`
The workspace looks like an older Brief layout. It can still be useful as context, but should not be treated as current Brief behavior.

#### `missing`
No `.brief/` exists.

#### `misconfigured`
A `.brief/` exists, but required current-schema paths or files are missing or broken.

#### `stale`
The schema is current, but the priorities artifact is stale relative to raw inputs or required human review state.

## C. `brief check --health` becomes the gateway command

Add:
```bash
brief check --health
brief check --health --json
```

This is the first command automation should run before trusting Brief.

### Text output contract
Human-readable, blunt, no magic.
Example:
```text
health: legacy-schema
- Legacy brief layout detected.
- Missing current-schema marker: rules/BUILD.md
- Missing current-schema marker: PRIORITIES-HUMAN.md
- Legacy context can still be fetched, but do not treat this as full modern Brief steering.
```

### JSON output contract
Example:
```json
{
  "state": "legacy-schema",
  "schema": "legacy",
  "checkedAt": "2026-04-08T08:00:00.000Z",
  "reasons": [
    "Legacy brief layout detected.",
    "Missing current-schema marker: rules/BUILD.md"
  ],
  "missingPaths": [
    "rules/BUILD.md",
    "PRIORITIES-HUMAN.md"
  ],
  "presentPaths": [
    "priorities.md",
    "priorities-raw.md"
  ]
}
```

### Exit codes
Recommended contract:
- `0` healthy-current-schema
- `3` missing
- `4` misconfigured
- `5` stale
- `6` legacy-schema

## D. `brief fetch` must be health-aware

`brief fetch` should never pretend all is well when it is not.

### Desired behavior by state

#### healthy-current-schema
- fetch into `.brief/raw/`
- print convention-first next steps
- point to `brief check --enrichment`
- if stale later, tell the agent to read `rules/BUILD.md` + `raw/` and update `priorities.md`

#### legacy-schema
- fetch can still run if useful
- output must clearly say this is legacy context mode
- do not imply full modern Brief steering is available

#### missing
- fail clearly
- say to run `brief init --template startup`

#### misconfigured
- optionally allow best-effort fetch
- but print explicit broken-state warning
- do not imply trustworthiness

#### stale
- allow fetch
- make it clear enrichment still needs to happen after fetch

### Messaging rules
- no references to unsupported commands
- no fake success language
- no vague “next step” hooks that do not match the real workflow

## E. `brief check --enrichment` must be convention-first

If stale, the guidance should be:
- read `.brief/rules/BUILD.md`
- read `.brief/raw/`
- update `.brief/priorities.md`

Not:
- hidden command magic
- stale references to removed commands
- hand-wavy “build needed” output with no workflow attached

## F. Add `brief migrate`

This is the one extra command worth adding.

### Why
Without migration, every schema evolution leaves behind a swamp of almost-compatible workspaces.
That is how trust dies.

### Command shape
```bash
brief migrate --dry-run
brief migrate
```

### Responsibilities
- detect legacy workspace shape
- show proposed changes
- create missing current-schema files/dirs
- preserve existing human-authored content
- move or map old artifacts where safe
- never silently destroy data

### Output
Migration report should include:
- detected legacy signals
- files created
- files mapped
- files left untouched
- manual follow-up required

### Constraints
- migration should be idempotent
- dry-run should be the default recommendation in docs

## G. Keep markdown as the real artifact

Brief gets weaker when the CLI tries to do the agent’s job.

The intended loop should be:
1. `brief check --health`
2. `brief fetch`
3. `brief check --enrichment`
4. if stale, the agent reads `rules/BUILD.md` + `raw/`
5. the agent writes `priorities.md`

That is the product.

## H. Add a real fixture matrix

Most of Brief’s risk lives in workspace shape, not fancy algorithms.

Tests should cover at least:
- healthy current schema
- missing `.brief/`
- legacy schema
- misconfigured schema
- stale current schema
- healthy fetch guidance
- legacy fetch guidance
- stale enrichment guidance
- migration dry-run
- migration apply

If these fixtures pass, Brief is probably honest.
If they do not, it probably is not.

## I. Docs should teach trust boundaries

README and SPEC should explicitly say:
- Brief is a convention first
- the CLI is for fetch/check/migrate, not planning magic
- health must be checked before trust
- legacy workspaces are degraded context mode

## Proposed Rollout

### Phase 1
Already aligned with current direction:
- canonical schema manifest
- explicit health states
- `brief check --health`
- health-aware `fetch`
- convention-first stale guidance

### Phase 2
Next critical step:
- `brief migrate --dry-run`
- `brief migrate`
- fixture coverage for migration paths

### Phase 3
Tightening:
- docs/spec cleanup
- remove any remaining stale wording
- ensure all automation examples start with health check

## Acceptance Criteria

This work is successful when:

1. A legacy workspace can no longer look healthy by accident.
2. `brief check --health` gives a stable, machine-readable answer.
3. `brief fetch` never implies trust when the workspace is missing, broken, or legacy.
4. `brief check --enrichment` points to the markdown workflow, not command magic.
5. Migration from legacy to current schema is supported and testable.
6. The CLI remains small.
7. Agents can reliably answer: “Should I trust Brief here, and how much?”

## Blunt Product Take

Brief gets better by becoming less clever.

It should not charm the user into thinking everything is okay.
It should not hide schema drift.
It should not grow a pile of helper commands to paper over uncertainty.

It should do four things well:
- define the convention
- tell the truth about health
- fetch raw state
- help agents trust or distrust the workspace correctly

That version of Brief has a shot.
The magical version does not.
